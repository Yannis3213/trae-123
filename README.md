# 跨境电商 - 月底集中处理跨境订单系统

基于 **React + Rsbuild + Go + Fiber + SQLite** 的跨境电商订单集中处理平台。

## 功能特性

- **三角色协作流程**：运营专员 → 仓配主管 → 店铺负责人
- **三阶连续办理**：商品刊登 → 库存同步 → 订单履约
- **列表分组统计**：待提交、已退回、重新提交
- **批量处理**：逐条校验、逐条返回成功/失败原因
- **提交校验**：角色、处理人、状态、版本、必填证据五重拦截
- **到期预警**：正常 / 临期（48h内） / 逾期，节点显示责任人
- **全程留痕**：处理记录、审计备注、异常原因、附件材料
- **角色切换**：登录后可快速切换身份演示完整流程

## 端口约定

所有端口通过环境变量统一配置，**前端 API 地址、后端监听端口、CORS 白名单、启动命令共用同一组变量**，避免写死。

| 变量 | 默认值 | 说明 |
|---|---|---|
| `FRONTEND_PORT` | 3000 | 前端端口 |
| `BACKEND_PORT` | 8080 | 后端端口 |

如需修改端口，请同时在两个 `.env` 文件中保持一致。

## 快速启动

### 前置条件

- Go ≥ 1.21
- Node.js ≥ 18
- npm / pnpm / yarn

### 1. 启动后端

```bash
cd backend
cp .env.example .env
# 如需修改端口请编辑 .env
go mod download
go run main.go
```

后端启动后：
- 监听端口由 `BACKEND_PORT` 控制（默认 `:8080`）
- **SQLite 自动初始化**：首次启动自动创建 `cross_border.db` 并写入演示数据
- CORS 白名单自动设置为 `http://localhost:${FRONTEND_PORT}`

### 2. 启动前端

```bash
cd frontend
# 如果要修改端口，可先设置环境变量：
# export FRONTEND_PORT=3000 BACKEND_PORT=8080
npm install
npm run dev
```

前端启动后访问 `http://localhost:${FRONTEND_PORT}`（默认 http://localhost:3000）。

### 3. 一键启动（两个终端）

```bash
# 终端1 - 后端
cd backend && go run main.go

# 终端2 - 前端
cd frontend && npm run dev
```

## SQLite 数据模型

首次启动自动建表并写入 10 条跨境订单演示数据，覆盖三阶段全流程。

| 表名 | 说明 |
|---|---|
| `users` | 用户账号（三角色演示账号） |
| `cross_border_orders` | 跨境订单（含状态、版本、环节、到期时间、三阶段数据） |
| `order_attachments` | 订单附件（按环节关联） |
| `processing_records` | 处理记录（状态流转、操作人、异常标记） |
| `audit_notes` | 审计备注（退回原因、补正要求等） |
| `exception_logs` | 异常原因（含异常类型、补正动作、解决状态） |

## 四类异常样例

系统预置了以下异常场景的演示数据，便于测试：

### 1. 缺材料 - 商品刊登环节

- **订单**：`CB20240601003`（便携咖啡机，德国）
- **现状**：运营专员提交后被仓配主管退回
- **异常原因**：缺少商品规格参数表、缺少高清白底图（≥5张）、缺少欧盟CE认证
- **测试路径**：
  1. 以 运营专员 登录
  2. 找到「已退回」分组中的订单
  3. 进入详情，补充完整材料后点击「重新提交」
  4. 切换 仓配主管，可再次审核

### 2. 超时或逾期 - 库存同步环节

- **订单**：`CB20240601006`（运动智能手环，加拿大）
- **现状**：库存同步已逾期，版本 v3，第 1 次重新提交
- **预警显示**：详情页和列表中显示红色「已逾期」标签，责任人显示在列表中
- **测试路径**：
  1. 以 运营专员 登录，处理该逾期订单
  2. 或使用 批量处理 对逾期订单操作，观察逐条拦截逻辑

### 3. 退回补正 - 订单履约环节

- **订单**：`CB20240601009`（进口婴幼儿奶粉，荷兰）
- **现状**：店铺负责人退回，缺少质检报告和原产地证明
- **异常类型**：`missing_evidence`
- **测试路径**：
  1. 运营专员 补充履约数据（物流、报关、质检报告等）
  2. 点击「重新提交」，异常记录自动标记为已解决
  3. 切换 店铺负责人 审核确认

### 4. 状态冲突 / 版本冲突

- **测试方法**：打开同一订单的两个浏览器标签页
- **步骤**：
  1. 在标签A提交操作成功（版本自动 +1）
  2. 在标签B使用旧版本号尝试提交
  3. 后端返回：`版本冲突：当前版本为 X，您提交的版本为 Y，请刷新后重试`
- **其他冲突**：越权操作（非当前处理人）、错误状态流转（如对已完成订单提交）、跨角色操作（如仓配主管审核履约环节）均会被拦截

## 核心校验规则（后端拦截）

| 校验维度 | 规则 | 错误示例 |
|---|---|---|
| **角色权限** | 提交只能运营专员；刊登/库存审核只能仓配主管；履约审核只能店铺负责人 | `只有运营专员可以提交订单` |
| **当前处理人** | 必须 `current_handler_id == 当前用户ID` | `该订单当前处理人为其他用户，您无权操作` |
| **状态流转** | pending→submitted→approved/returned，不符合顺序全部拦截 | `当前状态 X 不允许提交操作` |
| **版本号** | 提交版本必须等于数据库当前版本 | `版本冲突：当前版本为 X，您提交的版本为 Y` |
| **必填证据** | 各阶段数据字段不能为空且达到最小长度 | `商品刊登信息过于简略，需要包含标题、描述、图片等完整信息` |
| **批量逐条拦截** | 批量操作不会整批放行，每个订单独立校验并返回结果 | 返回数组中 `success:false` 附带具体失败原因 |

## 批量处理行为

- 入口：订单列表 → 勾选多个订单 → 顶部出现批量按钮
- 提交审核 / 审核通过 / 退回补正 均支持批量
- **关键保证**：即使批量中部分订单校验失败，已通过的也会正常提交，每条结果独立返回
- 批量退回时填写的原因会应用到所有选中订单，但可在详情中单独调整
- 批量结果展示：成功/失败计数 + 逐条明细（含失败原因）

## 到期预警规则

- **正常**：距到期 > 48 小时，绿色标签
- **临期**：距到期 ≤ 48 小时，黄色标签
- **逾期**：已超过到期时间，红色标签
- 每阶段独立计算到期时间（`listing_due_at` / `inventory_due_at` / `fulfillment_due_at`）
- 列表可按预警级别筛选，详情页 Steps 每个节点旁显示对应预警

## 主要 API

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/auth/login` | 登录 |
| GET | `/api/auth/me` | 当前用户 |
| GET | `/api/orders` | 订单列表（支持 group/stage/status/warning/search 过滤） |
| GET | `/api/orders/:id` | 订单详情（含附件/记录/备注/异常） |
| POST | `/api/orders/:id/action/:action` | 订单操作（submit/resubmit/approve/return） |
| POST | `/api/orders/batch` | 批量处理，逐条返回 |
| POST | `/api/orders/:id/audit-notes` | 添加审计备注 |
| GET | `/api/orders/statistics` | 汇总统计 |

## 开发说明

- 修改端口：同时修改 `backend/.env` 和设置前端启动环境变量
- 前端 `API_BASE_URL` 由 Rsbuild 在编译时根据 `BACKEND_PORT` 注入，无需硬编码
- 后端 CORS 白名单根据 `FRONTEND_PORT` 动态生成，保持一致即可跨域通信
- 删除 `backend/cross_border.db` 后重新启动后端可重置所有数据
