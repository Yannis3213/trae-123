# 物业服务中心 - 月底集中处理报修工单系统

基于 **Solid.js + Vite**（前端）和 **Python + Starlette**（后端）开发的物业服务中心月底集中处理报修工单系统。数据库使用本地 SQLite。

## 端口约定

本系统前后端共用一组端口变量，**请始终通过环境变量设置**，不要写死：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `FRONTEND_PORT` | 5173 | 前端 Vite 开发服务器端口 |
| `BACKEND_PORT` | 8000 | 后端 Starlette uvicorn 端口 |

前后端所有需要端口的地方（Vite 配置、后端监听、CORS 白名单、API 代理、README 启动命令）都读取这两个变量，保证一致。

## 启动方式

### 0. 环境变量准备（可选）

如需自定义端口，可在两个终端分别设置：

```bash
# macOS / Linux
export FRONTEND_PORT=5173
export BACKEND_PORT=8000
```

### 1. 启动后端

```bash
cd backend
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
python main.py
```

启动后：
- 后端监听 `http://localhost:<BACKEND_PORT>`（默认 8000）
- **SQLite 自动初始化**：首次启动时 `backend/repair_orders.db` 自动创建，表结构自动建立
- **演示数据自动载入**：包含正常、缺材料、超时、退回补正/状态冲突 4 类测试工单
- 健康检查：`GET http://localhost:<BACKEND_PORT>/api/health`

### 2. 启动前端（另开一个终端）

```bash
cd frontend
npm install
npm run dev
```

启动后访问 `http://localhost:<FRONTEND_PORT>`（默认 5173）。

## SQLite 说明

- **位置**：`backend/repair_orders.db`
- **初始化**：首次启动后端时自动建表（`init_db()`），无需手动操作
- **重置数据**：删除 `backend/repair_orders.db` 后重启后端即可重新生成含演示数据的干净库
- 主要数据表：
  - `users`：用户账号
  - `repair_orders`：报修工单（含状态、版本、当前处理人、到期预警标记）
  - `attachments`：证据附件
  - `processing_records`：处理记录/操作轨迹
  - `audit_notes`：审计备注（异常拦截、复核、补正、超期）
  - `exception_reasons`：异常原因（是否已解决、由谁解决）

| 角色 | 账号 username | 显示名 | 职责 | 可操作动作 |

| 报修审核主管 | `supervisor` | 王主管（维修主管） | 处理中段 | 开始处理、转办、退回补正、完成维修、回访确认 |

演示库里预置了 8 张工单，覆盖以下四类异常场景，可直接体验拦截逻辑：

### 1️⃣ 正常流转（可测试完整流程）
- **WX20260601001 - 1号楼3单元电梯异响**：已归档的完整流程工单，可查看完整处理记录、附件、审计轨迹

### 2️⃣ 缺材料 / 缺项（可测试补正流程）
- **WX20260601002 - 2号楼5层走廊灯不亮**：当前状态"退回补正"，业主信息、地址、描述全空。
  - 用 **李管家（登记员）** 进详情 → 点"补正信息"→ 填入缺失内容 → 保存补正 → 再点"派单"
  - 或用 **王主管（审核主管）** 对已派单工单执行"退回补正"
- **WX20260601007 - 小区儿童游乐设施螺丝松动**：状态"已补正"，可看到已解决的异常记录

### 3️⃣ 超时 / 逾期（可测试预警与批量推进拦截）
- **WX20260601003 - 3号楼201室卫生间漏水**：状态"处理中"，已逾期（红色行高亮，逾期徽标），系统自动标记责任处理人为王主管
- **WX20260601004 - 5号楼门禁系统故障**：状态"已回访"，临期预警（黄色行，临期徽标）
- **WX20260601006 - 4号楼1502室窗户密封条老化**：状态"复核中"，临期预警

### 4️⃣ 退回补正 / 状态冲突（可测试越权与状态拦截）
- 用 **登记员** 对状态"处理中"的工单尝试复核 → 后端拦截，写入审计备注
- 用 **复核负责人** 对状态"待分派"的工单尝试归档 → 后端拦截，写入审计备注
- 批量处理时故意选择不同状态工单 → 逐条返回成功/失败原因

## 核心功能说明

### 角色与权限
- 列表、详情、按钮、后端接口均按角色过滤和校验
- 后端每个操作同时校验：当前角色、当前处理人、状态流转合法性、数据版本、必填证据
- 直接调接口（绕过前端按钮）也会被拦截，并写入审计备注

### 列表与统计
- 列表按 **待分派 / 已转办 / 已回访** 三大分组统计
- 到期预警：**正常（绿）/ 临期 24h 内（黄）/ 逾期（红）** 三色区分，行背景同步染色
- 支持按分组、状态、来源模块筛选，支持关键字搜索

### 工单详情
- 保留**上一处理人意见**（黄色提示条）
- 支持**补正**：业主报修/维修派单缺项时，详情页可直接编辑保存
- **四个 Tab**：处理记录（操作轨迹）、附件、异常原因、审计备注（拦截追溯）
- 办理操作区根据角色 + 当前状态动态显示可执行动作

### 批量处理与结果
- 列表可多选工单，点击"批量处理"选择动作
- 批量结果**逐条列出成功/失败及失败原因**，共成功 X 条 / 失败 Y 条

### 后端拦截逻辑（写入审计备注）
- 🚫 越权操作：角色无该动作权限
- 🚫 重复提交：同版本重复提交
- 🚫 跨角色操作：例如登记员去处理中段工单
- 🚫 不符合顺序的状态变更：例如从"待分派"直接跳到"已完成"
- 🚫 版本冲突：他人已更新，需刷新后重试
- 🚫 缺证据：完成/回访/归档时要求上传证据
- 🚫 状态冲突：已归档工单不可再操作

所有拦截均写入 `audit_notes`，详情页"审计备注"Tab 中可追溯。

## 刷新数据一致性

列表、详情页所有数据均实时从 SQLite 读取，刷新页面后：
- 列表统计、到期预警状态
- 详情字段、附件、处理记录
- 审计轨迹、异常原因

均与数据库保持一致。
