# 社区卫生服务中心 - 慢病随访单管理系统

## 项目概述

社区卫生服务中心月底集中处理慢病随访单系统，实现导诊护士、全科医生、医务科主任三级角色协作，
支持慢病档案管理、用药提醒、随访单登记三大业务模块，具备完整的状态流转、
审计追踪、异常拦截和版本控制机制。

## 技术栈

- **前端**: Vue 3 + Vite + Element Plus + Pinia + Vue Router
- **后端**: Node.js + Hono
- **数据库**: SQLite (better-sqlite3)
- **认证**: JWT
- **端口**: 前端 3107，后端 8107

## 端口配置

> ⚠️ 所有端口配置已统一，请勿修改：
> - 前端服务端口: **3107**
> - 后端服务端口: **8107**
> - CORS 白名单: `http://localhost:3107`, `http://127.0.0.1:3107`
> - 前端代理: `/api` -> `http://localhost:8107`

## 快速开始

### 1. 安装依赖

```bash
# 安装后端依赖
cd backend
npm install

# 安装前端依赖
cd ../frontend
npm install
```

### 2. 初始化数据库

```bash
cd backend
npm run init-db
```

初始化脚本会创建：
- 7 张数据库表（随访单、附件、处理记录、审计备注、异常原因、慢病档案、用药提醒）
- 3 个演示账号
- 6 条测试随访单（覆盖四类业务场景）
- 3 条慢病档案
- 2 条用药提醒

### 3. 启动服务

```bash
# 启动后端服务 (端口 8107)
cd backend
npm run dev

# 启动前端服务 (端口 3107)
cd ../frontend
npm run dev
```

### 4. 访问系统

打开浏览器访问: `http://localhost:3107`

## 异常样例与拦截机制

### 1. 越权访问拦截
- **场景**: 导诊护士尝试访问主任审核中的单据
- **后端验证**: `requireRole` 中间件检查角色权限
- **响应**: 403 权限不足

### 2. 重复提交拦截
- **场景**: 两个医生同时打开同一份待处理单据
- **后端验证**: 检查 `current_handler_id` 字段
- **响应**: 409 该单据已被其他医生处理

### 3. 状态冲突拦截
- **场景**: 尝试将「草稿」直接变更为「主任审核中」
- **后端验证**: `validateStatusTransition` 检查状态流转合法性
- **响应**: 400 状态流转不合法

### 4. 旧版本提交拦截
- **场景**: 使用过时的页面版本提交（后端版本已递增）
- **后端验证**: `validateVersion` 比较客户端版本与数据库版本
- **响应**: 409 版本冲突，请刷新后重试

### 5. 缺证据拦截
- **场景**: 提交「待提交」状态但未上传随访单附件
- **后端验证**: `validateEvidence` 检查必要证据类型
- **响应**: 400 缺少必要证据: followup_form

### 6. 逾期批量推进拦截
- **场景**: 批量处理包含逾期单据
- **后端验证**: `checkOverdue` 检查到期日期，逾期单据逐条拦截
- **响应**: 单条失败，原因: 已逾期X天，需逐单处理，不得批量推进

## 核心功能

### 业务区模块
1. **慢病档案** - 查询、新增、补正患者慢病档案
2. **用药提醒** - 查询、新增、编辑、删除用药提醒
3. **慢病随访单登记** - 快速查看近期登记记录和统计

### 状态流转
```
草稿 → 待提交 → 医生处理中 → 主任审核中 → 已完成 → 已归档
         ↓         ↓            ↓
      已退回 ← 已退回 ←      已退回
         ↓
      重新提交 → 医生处理中 → ...
```

### 审计追踪
- 所有操作均记录审计日志（create/update/submit/process/review/complete/return/archive）
- 状态变更、异常原因、处理意见完整记录
- 审计备注可追溯「待提交」「已退回」「重新提交」等关键节点

### 到期预警
- **正常**: 到期日 > 3 天（绿色）
- **临期**: 到期日 ≤ 3 天（黄色）
- **逾期**: 到期日 < 今天（红色）
- 节点超时显示当前责任人

## 数据库表结构

### followup_forms - 慢病随访单主表
- 核心字段：id, patient_name, id_card, status, current_role, current_handler_id, version
- 状态字段：submitted_at, doctor_processed_at, director_reviewed_at, completed_at, returned_at, archived_at

- 证据类型: followup_form(随访单), vital_signs(生命体征), medication_record(用药记录), treatment_plan(治疗方案)

- 记录所有操作动作、备注、扩展数据（JSON）

- 记录所有拦截异常：类型、原因、操作人

### chronic_records - 慢病档案表
### medication_reminders - 用药提醒表

## 关键技术实现

### 1. 版本控制
每次状态变更时 `version` 字段自动 +1，提交时比较客户端版本与后端版本，
不一致则拒绝提交并返回 409 状态码。

### 2. 证据校验
不同状态节点要求不同的证据附件，提交前校验证据完整性，
缺少必要证据则拦截并返回具体缺失项。

### 3. 权限判断
状态变更时同时校验：
- 当前用户角色是否允许操作该状态
- 状态流转是否符合预设规则
- 当前处理人是否匹配（避免重复处理）

### 4. 异常拦截
所有被拦截的异常操作都会：
- 写入 `exception_reasons` 表记录原因
- 写入 `audit_logs` 表便于追溯
- 返回明确的错误信息给前端

### 5. 批量处理
- 逐条校验，逐条返回结果
- 逾期单据自动拦截，不得整批放行
- 成功/失败原因清晰展示
- 详情页留下补正动作和异常原因

## 刷新一致性保证

所有数据变更均通过同一数据库实例操作，刷新页面后：
- 列表数据与后端一致
- 详情页状态与列表一致
- 统计数据实时计算
- 操作记录完整显示
