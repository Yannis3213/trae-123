# 养老护理院 - 月底集中处理护理计划单系统

院区主任复核视角的护理计划单集中处理系统，支持多角色协作、状态流转、批量处理与审计追踪。

---

## 端口配置（全局统一）

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 (Nuxt 3) | **3004** | `http://localhost:3004` |
| 后端 (Rust Rocket) | **8004** | `http://localhost:8004` |

> ⚠️ 前端请求地址、后端监听端口、CORS 白名单全部使用上述端口，不可变更。

---

## 角色说明

| 角色 | 职责 | 队列视角 |
|------|------|----------|
| **护理计划登记员** | 发起护理计划单、退回补正 | 待派发、处理中 |
| **护理计划审核主管** | 办理审核、提交复核 | 处理中 |
| **养老护理院复核负责人** | 最终复核、归档关闭 | 处理中、已关闭 |

状态流转：`待派发 → 处理中 → 已关闭`，支持退回补正（逆向）。

---

## 启动命令

### 1. 后端启动

```bash
cd backend
cargo run
```

后端将在 `http://localhost:8004` 启动，首次启动自动初始化 SQLite 数据库。

### 2. 前端启动

```bash
cd frontend
npm install
npm run dev
```

前端将在 `http://localhost:3004` 启动。

---

## SQLite 初始化说明

数据库文件路径：`backend/nursing_home.db`

首次启动后端时自动执行初始化：
1. 创建 5 张核心表：`care_plans`、`attachments`、`processing_records`、`audit_notes`、`exception_reasons`
2. 写入演示账号数据
3. 预置 6 条样例护理计划单（覆盖正常、缺材料、超时、退回补正、状态冲突场景）

如需重置数据库：

```bash
rm backend/nursing_home.db && cd backend && cargo run
```

## 四类异常样例

系统对以下异常场景进行拦截并记录审计日志：

### 异常 1：越权操作
- **场景**：登记员尝试执行复核归档操作
- **触发**：使用 `registrar` 账号调用 `/api/care-plans/{id}/review` 接口
- **返回**：`403 Forbidden` - `当前角色无权限执行该操作`
- **审计记录**：前后状态不变，失败原因记录越权详情

### 异常 2：状态冲突
- **场景**：计划单已关闭，仍尝试提交审核
- **触发**：对状态为 `已关闭` 的计划单调用 `/api/care-plans/{id}/submit` 接口
- **返回**：`409 Conflict` - `当前状态不允许该操作，期望状态：处理中`
- **审计记录**：记录状态冲突，标注期望状态与实际状态

### 异常 3：旧版本提交（乐观锁）
- **场景**：两人同时编辑，后提交者使用了旧版本号
- **触发**：提交请求时 `version` 字段小于数据库当前版本
- **返回**：`409 Conflict` - `版本冲突，请刷新页面后重试`
- **审计记录**：记录提交版本号与当前版本号差异

### 异常 4：缺少必填证据
- **场景**：家属确认环节未上传签字附件
- **触发**：提交时 `family_confirmed` 为 true，但 `attachments` 中无家属签字文件
- **返回**：`400 Bad Request` - `缺少必填证据：家属签字确认单`
- **审计记录**：记录缺失的证据项清单

## 月底到期预警

队列自动分为三队，不混排：

| 预警级别 | 判定规则 | 视觉标识 |
|----------|----------|----------|
| 🟢 正常 | 距截止日 > 3 天 | 绿色标签 |
| 🟡 临期 | 距截止日 ≤ 3 天 且 ≥ 0 天 | 黄色标签 |
| 🔴 逾期 | 当前日期 > 截止日 | 红色标签，自动计算到责任人 |

逾期批量推进时逐条拦截，每条返回独立的成功/失败原因。
