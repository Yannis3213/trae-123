# 口腔连锁门诊-月底集中处理治疗计划单系统

基于 **Solid.js + Vite + Node.js + NestJS + SQLite** 构建的口腔门诊治疗计划单月底集中处理系统。

---

## 系统端口约定（全局统一）

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端（Vite Dev Server） | **3002** | 浏览器访问地址：http://localhost:3002 |
| 后端（NestJS） | **8002** | API 地址：http://localhost:8002/api |
| CORS 白名单 | - | 后端仅允许 http://localhost:3002 跨域请求 |
| 前端 API BASE_URL | - | 固定请求 http://localhost:8002/api |

> ⚠️ 所有端口配置已在代码中统一，请勿修改。

---

## 技术栈

- **前端**：Solid.js 1.9 + Vite 6 + TypeScript + @solidjs/router
- **后端**：NestJS 10 + TypeORM + better-sqlite3 + class-validator
- **数据库**：本地 SQLite（无需额外安装服务）
- **认证**：基于 Bearer Token 的模拟鉴权（Token = 用户 ID）

## 快速启动

### 第一步：启动后端服务

```bash
cd backend
npm install
npm run start:dev
```

启动后：
- 服务监听：http://localhost:8002
- SQLite 数据库文件自动创建于：`backend/data/app.db`
- 首次启动自动执行**种子数据初始化**（用户 + 4 张演示治疗计划单）
- 看到 `Backend is running on http://localhost:8002` 和 `[Seed] 数据初始化完成` 即为启动成功

### 第二步：启动前端服务（新开终端）

```bash
cd frontend
npm install
npm run dev
```

启动后：
- 浏览器访问：http://localhost:3002
- 看到 Vite 输出 `Local: http://localhost:3002/` 即为启动成功

---

## SQLite 数据库说明

- **数据库路径**：`backend/data/app.db`（首次启动自动创建）
- **自动建表**：通过 TypeORM `synchronize: true` 自动创建所有表
- **自动播种**：`AppModule.onModuleInit` 中检测用户表为空时自动执行种子脚本
- **重置数据**：删除 `backend/data/app.db` 文件后重启后端即可重新初始化

### 包含的数据表

| 表名 | 说明 |
|------|------|
| user | 用户（含角色） |
| treatment_plan | 治疗计划单（主表） |
| attachment | 附件（患者档案/治疗计划/复诊提醒三类） |
| process_record | 处理记录（状态流转审计链） |
| audit_note | 审计备注 |
| exception_cause | 异常原因（材料/权限/时限/状态） |

登录页面可直接选择角色，自动填充对应账号：

| 角色 | 账号 | 密码 | 姓名 | 可处理状态 |
|------|------|------|------|-----------|
| 前台顾问 | `consultant1` | `123456` | 李顾问 | 待确认 → 已确认 |
| 口腔医生 | `doctor1` | `123456` | 张医生 | 已确认 → 异常/待复查；异常 → 待复查 |
| 门店院长 | `dean1` | `123456` | 王院长 | 待复查 → 已复查 → 已归档 |

### 状态流转主链路

```
待确认(pending_confirm)
    │ 前台顾问 confirm
    ▼
已确认(confirmed)
    │ 口腔医生 mark_exception（需证据+异常原因）
    ▼
异常(exception) ──┐
    │ 口腔医生 resolve_exception
    ▼               │
待复查(pending_review) ◄──┘
    │ 门店院长 review
    ▼
已复查(reviewed)
    │ 门店院长 archive
    ▼
已归档(archived)
```

> 口腔医生不能替门店院长归档，前台顾问不能跳环节处理，后端严格校验。

## 核心功能

### 1. 角色切换
- 顶部导航栏显示当前角色与姓名，点击"切换用户"返回登录页重新选择角色

### 2. 列表筛选
- 按状态筛选：待确认/已确认/异常/待复查/已复查/已归档
- 按到期预警筛选：全部/正常/临期（7天内）/逾期
- 关键词搜索：患者姓名、身份证、手机号、计划单号

### 3. 详情办理
- 顶部信息卡：单号、患者、状态、版本号、截止日期
- 三个模块标签页：患者档案 / 治疗计划 / 复诊提醒，各自显示附件列表和补正按钮
- 办理区域：根据当前角色+状态动态显示可操作按钮
- 异常原因区：按材料/权限/时限/状态四类分组展示
- 处理历史时间线：完整审计链（操作人、动作、状态变化、意见、证据、时间）
- 审计备注区：可追加备注，所有历史备注可见

### 4. 补正功能
- 患者档案或治疗计划缺项时，可直接从详情页补正：上传附件、勾选完成状态

### 5. 批量处理
- 列表多选后点击批量操作按钮
- 统一填写办理意见后提交
- 返回结果**逐条显示成功/失败原因**（如：版本冲突、非当前处理人、状态不匹配等）

### 6. 到期预警
- 分三栏看板：正常 / 临期（7天内） / 逾期，三队不混排
- 每张卡片显示责任人、截止日期、剩余天数

### 7. 统计概览
- 各状态数量统计卡片
- 到期预警统计柱状图

## 后端安全校验（直接调接口也会拦截）

每次办理操作后端都会执行以下校验，不依赖前端按钮：

| 校验项 | 说明 | 拦截场景 |
|--------|------|----------|
| **角色权限** | 每个状态只能由指定角色处理 | 越权操作（如医生直接归档） |
| **当前处理人** | 必须是计划单的 currentHandler | 非指定处理人操作 |
| **状态流转** | 只能按主链路流转 | 跳环节、非法流转 |
| **版本乐观锁** | 前端 version 必须等于数据库 version | 重复提交、旧版本提交、并发冲突 |
| **证据校验** | 异常必须有附件/证据；复查必须有复诊提醒证据 | 缺证据请求 |
| **必填信息** | 复核时校验患者档案/治疗计划/复诊提醒完整性 | 信息缺项拦截 |

## API 接口一览（前缀 /api）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/auth/login` | 登录，返回用户信息 + token |
| GET | `/users` | 用户列表 |
| GET | `/treatment-plans` | 列表（?status=&deadlineWarning=&search=） |
| GET | `/treatment-plans/:id` | 详情（含附件/记录/备注/异常） |
| POST | `/treatment-plans/process` | 办理单条（含所有校验） |
| POST | `/treatment-plans/batch-process` | 批量办理（逐条返回结果） |
| POST | `/treatment-plans/audit-note` | 添加审计备注 |
| POST | `/treatment-plans/correct` | 补正模块材料/信息 |
| GET | `/statistics` | 各状态数量 + 到期预警统计 |
| GET | `/statistics/due-warning` | 到期预警三栏数据 |

## 数据一致性保证

- 所有写操作通过 **TypeORM 事务** 执行（治疗计划单 + 处理记录 + 附件 + 异常原因 同事务提交/回滚）
- 版本号乐观锁防止并发覆盖
- 刷新后列表、详情、统计、操作记录完全一致（全部从 SQLite 实时读取）
