# 工程监理公司 - 月底集中处理旁站记录单系统

## 一、系统概述

本系统为工程监理公司打造的「月底集中处理旁站记录单」数字化办理平台，覆盖旁站记录单从登记、过程核验到复核归档的完整办理链路。系统按照岗位拆开页面视角，支持角色切换、列表筛选、详情办理、批量处理，并提供到期预警和完整台账。

### 技术栈
- **前端**：Remix + React + Tailwind CSS
- **后端**：Node.js + Koa 2
- **数据库**：本地 SQLite（better-sqlite3）
- **端口**（从环境变量派生，未设置时使用默认值）：
  - 前端：环境变量 `FRONTEND_PORT`，默认 `5173`
  - 后端：环境变量 `BACKEND_PORT`，默认 `3001`
  - 所有配置均从 `backend/src/config/index.js` 和 `frontend/vite.config.js` 的环境变量派生：
    - 后端监听端口
    - CORS 白名单
    - 前端 Vite 代理（`/api` → 后端端口）
    - README 启动说明
  - ⚠ 如需修改端口，请通过设置环境变量，不要直接写死常量

## 二、快速启动

### 1. 安装依赖

```bash
# 方式一：一键安装所有依赖
cd trae-123-5
npm run install:all

# 方式二：分步安装
npm install
cd backend && npm install
cd ../frontend && npm install
```

### 2. 初始化数据库 + 演示数据

```bash
# 初始化表结构
npm run init:db

# 导入演示数据（会先清空再重建）
npm run seed
```

### 3. 启动服务

```bash
# 方式一：同时启动前后端（推荐）
npm run dev

# 方式二：分别启动
# 启动后端
npm run start:backend
# 启动前端（另一个终端）
npm run start:frontend
```

**自定义端口（可选）**：

端口通过环境变量控制，无需修改代码：

```bash
# macOS / Linux
FRONTEND_PORT=8080 BACKEND_PORT=4000 npm run dev

# Windows PowerShell
$env:FRONTEND_PORT=8080; $env:BACKEND_PORT=4000; npm run dev
```

启动完成后访问（根据端口配置调整）：
- 前端：`http://localhost:${FRONTEND_PORT:-5173}`
- 后端健康检查：`http://localhost:${BACKEND_PORT:-3001}/api/health`

## 四、演示数据说明

执行 `npm run seed` 后会创建 **7 条旁站记录演示数据**，覆盖四类典型场景：

### 1. 正常流转样例
- **单据**：地下车库主体混凝土浇筑
- **状态**：`已同步`（synced）
- **用途**：验证完整流转链路：登记→提交→审核通过→同步归档

### 2. 缺材料样例
- **单据**：钢筋绑扎工程旁站
- **状态**：`缺料`（material_missing）
- **异常原因**：缺少检查记录和签字证据
- **用途**：验证「缺料退回 → 补正提交」流程，以及必填证据拦截

### 3. 超时/逾期样例
- **单据**：防水工程旁站
- **状态**：`逾期`（overdue）
- **预警分组**：逾期
- **异常原因**：超出整改期限 3 天仍未完成复核
- **用途**：验证到期预警分组、节点超时落到责任人、逾期批量推进逐条拦截

### 4. 退回补正样例
- **单据**：模板安装旁站
- **状态**：`退回补正`（returned）
- **预警分组**：临期
- **异常原因**：模板垂直度检测数据填写不规范
- **用途**：验证「退回补正 → 监理员补正提交」链路

### 5. 状态冲突样例
- **单据**：砌体工程旁站
- **状态**：`状态冲突`（status_conflict）
- **用途**：验证异常停原队列、原因同步到页面和接口

### 6. 待审核样例
- **单据**：脚手架搭设旁站
- **状态**：`待审核`（pending_review）
- **用途**：验证专业监理工程师审核入口

### 7. 审核通过待归档样例
- **单据**：基坑支护旁站
- **状态**：`审核通过`（review_passed）
- **用途**：验证总监代表复核归档入口

| 模块 | 路径 | 可见角色 | 说明 |
|------|------|----------|------|
| 旁站记录单登记 | `/registration` | 监理员 | 维护入口数据，新增/补正后提交 |

| 旁站记录台账 | `/ledger` | 全部角色 | 沉淀所有处理结果，支持按旁站记录线索筛选，追溯问题通知、整改复核状态 |
| 到期预警 | `/warning` | 全部角色 | 按正常/临期/逾期分组，节点超时落到责任人；批量推进逐条拦截 |

### 状态流转图

```
[创建] → 待审核 ──审核通过──→ 审核通过 ──同步归档──→ 已同步
                │                │
                ├──退回补正      ├──退回补正
                ├──缺料退回      ├──缺料退回
                ├──标记逾期      └──标记逾期
                └──状态冲突
```

### 完整办理链路

#### 链路一：正常流转（推荐测试路径）
```
监理员（登记）
    ↓ 填写：项目信息、旁站部位、三项证据（照片/记录/签字）
    ↓ 提交（自动版本号+1，currentHandlerId 移交至专业监理）
专业监理工程师（过程核验）
    ↓ 核对：证据完整性、异常原因
    ↓ 审核通过（自动版本号+1，currentHandlerId 移交至总监代表）
总监代表（复核归档）
    ↓ 确认：最终结果、整改复核状态
    ↓ 同步归档（版本号+1，currentHandlerId 清空为 null）
```

#### 链路二：缺料补正
```
监理员提交时证据不全
    ↓ 系统自动拦截 → 转入「缺料」状态
    ↓ 异常原因记入 abnormal_reasons 表
    ↓ currentHandlerId 移交回原登记员
监理员（登记模块）
    ↓ 看到缺料队列 → 补充缺失证据
    ↓ 重新提交 → 转入「待审核」
    ↓ currentHandlerId 移交至专业监理
```

#### 链路三：退回补正
```
专业监理审核时发现问题
    ↓ 选择「退回补正」→ 填写异常原因
    ↓ 转入「退回补正」状态
    ↓ currentHandlerId 移交回原登记员
监理员（登记模块）
    ↓ 看到退回队列 → 修改内容
    ↓ 重新提交 → 转入「待审核」
```

#### 链路四：逾期处理
```
专业监理标记逾期
    ↓ 填写异常原因 → 转入「逾期」状态
    ↓ currentHandlerId 移交至总监代表
总监代表（复核归档模块）
    ↓ 看到逾期队列 → 可退回补正或标记异常
    ↓ 批量处理时逐条拦截，每条独立校验
```

### currentHandlerId 移交规则

| 操作 | 目标状态 | 下一处理人 |
|------|----------|------------|
| 提交/补正提交 | `待审核` | 专业监理工程师（按角色查询第一个） |
| 审核通过 | `审核通过` | 总监代表（按角色查询第一个） |
| 退回补正 | `退回补正` | 原登记员 |
| 缺料退回 | `缺料` | 原登记员 |
| 标记逾期 | `逾期` | 总监代表（按角色查询第一个） |
| 状态冲突 | `状态冲突` | 当前操作人 |
| 同步归档 | `已同步` | `null`（清空） |

### SQLite 持久化规则

每次流转操作会同时写入以下表，刷新页面后数据一致：

| 表名 | 写入内容 |
|------|----------|
| `side_records` | 更新 `status`、`current_handler_id`、`version`、`abnormal_reason`、`abnormal_type` 等字段 |
| `process_records` | 新增一条处理记录，包含：`action`、`from_status`、`to_status`、`operator_id`、`handler_id`、`evidence_submitted`、`evidence_missing`、`abnormal_reason`、`remark`、`version` |
| `abnormal_reasons` | 异常操作（退回/缺料/逾期/状态冲突）新增一条异常原因记录，包含：`reason_type`、`reason_detail`、`related_field`、`reported_by` |
| `audit_notes` | 通过「添加备注」接口手动写入审计意见 |

## 六、核心校验规则（后端接口层）

即使绕开页面直调接口，后端也会基于以下维度拦截：

| 校验维度 | 说明 |
|----------|------|
| **当前角色** | 每个接口限定可访问角色，越权返回 403 |
| **当前处理人** | 单据必须分配给当前用户才能办理（`currentHandlerId === userId`） |
| **状态机** | 严格的状态流转规则，非法流转被拦截 |
| **版本号** | 提交时**必须**带上当前版本号，旧版本提交返回「版本冲突」；不传版本号直接拦截 |
| **必填证据** | 提交/审核通过/归档前校验：现场照片、检查记录、签字确认三项，缺失则拦截并记入异常原因 |

异常停原队列，原因同步到页面展示和接口返回。

## 七、前端可办队列刷新机制

| 模块 | 访问角色 | 过滤规则 | 刷新机制 |
|------|----------|----------|----------|
| 旁站记录单登记 | 监理员 | `状态 in [待审核, 退回补正, 缺料]` **且** `registrarId === 当前用户ID` | `refreshKey` 自增触发 useEffect 重新加载 |
| 过程核验 | 专业监理工程师 | `状态 in [待审核, 缺料, 逾期, 状态冲突]` **且** `currentHandlerId === 当前用户ID` | `refreshKey` 自增触发 useEffect 重新加载 |
| 复核归档 | 总监代表 | `状态 in [审核通过, 逾期, 已同步]`，已同步单据仅查看不可操作 | `refreshKey` 自增触发 useEffect 重新加载 |
| 详情弹窗 | - | 办理成功后自动刷新列表 + 刷新详情 | `onRefresh()` 回调 + `loadDetail()` 重载 |

## 八、批量处理规则

- 在任意列表页勾选多条单据 → 点击「批量处理」
- 系统逐条执行，每条独立校验【角色、处理人、状态、版本、证据】五维
- 结果页面展示：总条数、成功数、失败数、**每条单据的成功/失败明细、新状态、新处理人、说明**
- 逾期批量推进时同样逐条拦截，可回到详情做补正动作、查看异常原因
- 异常操作（退回/缺料/逾期）必须填写异常原因，否则前端拦截

## 九、SQLite 数据表

数据库文件位置：`backend/data/side_records.db`

| 表名 | 说明 |
|------|------|
| `users` | 用户账号（含角色、部门、联系方式） |
| `side_records` | 旁站记录单主表（含状态、版本、预警分组、异常原因、问题通知、整改复核状态等） |
| `attachments` | 附件表 |
| `process_records` | 处理流水记录（每条操作的动作、状态变化、证据、异常原因、版本） |
| `audit_notes` | 审计备注 |
| `abnormal_reasons` | 异常原因明细（可标记是否已解决） |

刷新页面后，列表、详情、统计、操作记录均与数据库一致。

## 十一、复查闭环测试清单

建议按以下步骤验证系统：

1. **正常流转**：用监理员登记 → 提交 → 专业监理审核通过 → 总监代表同步归档，检查每个节点状态、处理记录、版本号、currentHandlerId
2. **缺材料**：监理员提交时故意不填检查记录，观察是否被拦截并转入「缺料」状态，异常原因是否记录，currentHandlerId 是否移交回登记员
3. **超时/逾期**：查看到期预警页面，逾期单据的责任人、批量推进逐条拦截是否生效
4. **退回补正**：专业监理退回一条单据，检查监理员列表是否可见、补正后能否重新提交，currentHandlerId 是否正确移交
5. **直调接口越权测试**：用监理员 token 调用 `/api/side-records/:id/review`，预期返回 403
6. **旧版本提交**：记下某单据版本号，刷新后用旧版本号提交，预期返回「版本冲突」
7. **不传版本号测试**：直调接口故意不传 version，预期返回「参数缺失：必须提交当前版本号 version」
8. **台账追溯**：在台账页按旁站记录线索搜索，验证问题通知、整改复核状态列的展示
9. **批量处理测试**：勾选多条不同状态的单据批量操作，验证每条独立校验，成功/失败原因、新状态、新处理人逐条展示
10. **处理人移交验证**：提交/退回/缺料/逾期/归档各操作后，验证 currentHandlerId 是否按规则正确更新
