# 企业人事共享中心 - 月底集中处理入职办理单系统

## 项目概述

企业人事共享中心入职办理单集中处理系统，支持三角色接力办理、异常拦截、审计追踪。

- **前端**：TanStack Start (React)，端口 3005
- **后端**：Go + Chi 框架，端口 8005
- **数据库**：SQLite（内嵌，自动初始化）

## 端口配置

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 | 3005 | http://localhost:3005 |
| 后端 API | 8005 | http://localhost:8005/api |

前端请求地址、后端监听端口、CORS 白名单统一使用此端口配置。

## 快速启动

### 1. 启动后端

```bash
cd backend
go mod tidy
mkdir -p data
go run cmd/server/main.go
```

后端启动后会自动：
- 创建 SQLite 数据库文件 `data/onboarding.db`
- 初始化数据库表结构
- 插入演示数据（首次启动时）

### 2. 启动前端

```bash
cd frontend
npm install
npm run dev
```

访问 http://localhost:3005

## 核心流程

```
入职资料 → 合同签署 → 账号开通
  (登记员)   (审核主管)   (复核负责人)
```

### 状态流转

- **待派发** (pending)：登记员提交后，等待下一角色认领
- **处理中** (processing)：已被对应角色认领，正在办理
- **退回补正** (returned)：审核/复核不通过，退回登记员补正
- **已完成** (completed)：全部节点办理完成
- **已关闭** (closed)：异常关闭或手动关闭

## 到期预警

- **正常**：截止日期 > 3 天
- **临期**：截止日期 ≤ 3 天且未逾期
- **逾期**：已过截止日期

逾期节点显示责任人和异常原因。

## 异常拦截机制

### 后端校验项

1. **角色权限校验**：当前处理角色与操作用户角色不匹配时拒绝
2. **状态冲突校验**：已完成/已关闭单据无法再次操作
3. **版本冲突校验**：提交版本 != 当前版本时拒绝（防止重复提交/并发冲突）
4. **必填附件校验**：通过前检查当前节点必填附件是否齐全
5. **越权访问校验**：列表按角色过滤，非本角色单据仅可查看

### 批量处理

- 逐条校验、逐条返回结果
- 逾期批量推进不整批放行，每条独立拦截
- 成功/失败均附带具体原因
- 详情页留有补正动作和异常原因记录

## 审计备注

支持三种状态标签的审计备注：
- 待派发
- 处理中
- 已关闭

所有处理操作自动生成审计记录，可追溯完整操作历史。

| 批量异常 | 批量选择包含不同状态/角色的单据执行操作 |

## API 接口

### 认证
- `POST /api/auth/login` - 登录
- `GET /api/auth/me` - 获取当前用户

### 入职办理单
- `GET /api/orders/` - 列表（支持 status、node、search、role 参数）
- `GET /api/orders/:id` - 详情（含附件、处理记录、审计备注）
- `POST /api/orders/:id/process` - 办理操作（action: submit/claim/approve/return/close）
- `POST /api/orders/batch` - 批量处理
- `POST /api/orders/:id/audit-notes` - 添加审计备注

### 元数据
- `GET /api/meta` - 获取角色、状态、节点等枚举值

## 数据库表结构

- `users` - 用户表
- `onboarding_orders` - 入职办理单
- `attachments` - 附件
- `process_records` - 处理记录
- `audit_notes` - 审计备注

## SQLite 数据持久化

数据文件位置：`backend/data/onboarding.db`

如需重置数据，删除该文件后重启后端服务，系统会自动重新创建并加载 seed 数据。

## 穿透测试建议

1. **正常流转**：登记员提交 → 审核主管认领通过 → 复核负责人通过 → 完成
2. **缺材料**：在资料不齐时尝试通过，验证拦截
3. **超时逾期**：查看王五单据的逾期标记和异常原因
4. **退回补正**：审核主管退回赵六的单据，登记员补正后重新提交
5. **越权测试**：
   - 用薪酬主管（需自行理解为审核主管角色）尝试归档最后节点
   - 用人事专员（登记员）跳环节直接处理合同节点
6. **状态冲突**：用 API 对已完成单据重复提交
7. **旧版本提交**：传入低于当前版本的 version 参数
8. **缺证据请求**：在必填附件不全时调用 approve 接口
9. **批量穿透**：选择多张不同状态单据批量处理，验证逐条结果
