# 餐饮连锁总部-月底集中处理门店订货单系统

## 项目概述

本系统用于餐饮连锁总部月底集中处理门店订货单，解决订货单处理阶段容易丢失失败明细的问题。系统将入口、办理和复核串起来，通过**原料订货**、**到货验收**、**库存回写**三个业务模块驱动状态、证据和异常原因变化。

### 技术栈

- **前端**: Preact + Vite
- **后端**: Node.js + Express
- **数据库**: SQLite (better-sqlite3)

### 端口配置

⚠️ **端口配置全局统一，请勿修改其他端口**

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 | 3001 | `http://localhost:3001` |
| 后端 | 8001 | `http://localhost:8001` |
| CORS 白名单 | 3001 | `http://localhost:3001` |
| 前端请求地址 | /api | Vite 代理转发到后端 8001 端口 |
| 后端监听端口 | 8001 | 所有 API 监听此端口 |

## 业务流程

### 主链路状态流转

```
待确认 (pending_material)
    ↓ 门店店长补齐材料
待验收 (pending_acceptance)
    ↓ 品控专员验收通过
待复核 (pending_review)
    ↓ 营运经理复核通过
已完成 (completed)
```

### 角色边界

- **门店店长**: 原料订货模块，提交订货材料（需校验材料完整性）
- **品控专员**: 到货验收模块，验收补正（需校验验收时限）
- **营运经理**: 库存回写模块，复核收口（需校验回写证据）

### 筛选条件

队列列表支持筛选：
- **待确认**: `pending_material` + `pending_acceptance` + `pending_review`
- **异常**: `exception`
- **已复查**: `recheck_pending`

## 核心功能

### 前端功能

1. **角色切换**: 侧边栏支持快速切换角色，便于测试
2. **列表筛选**: 按状态、门店、关键词搜索
3. **详情办理**: 进入订单详情进行业务处理
4. **批量处理**: 选中多条订单进行逾期批量推进
5. **批量结果**: 逐条返回成功/失败原因

### 后端校验

提交时自动校验以下内容，不符合则拦截：

| 校验项 | 说明 |
|--------|------|
| 当前角色 | 校验操作人角色是否匹配当前处理节点 |
| 当前处理人 | 校验操作人是否为当前节点指定处理人 |
| 状态 | 校验当前订单状态是否允许该操作 |
| 版本号 | 校验提交版本是否与当前版本一致，防止重复提交 |
| 必填证据 | 校验是否上传了必填证据（发票、验收结果、回写凭证） |

被拦截的请求类型：
- 越权操作
- 重复提交
- 状态冲突
- 旧版本提交
- 缺证据请求

### 数据库表结构

1. **store_orders**: 门店订货单主表
2. **order_items**: 订货单明细
3. **attachments**: 附件（凭证）
4. **processing_records**: 处理记录（操作日志）
5. **audit_notes**: 审计备注
6. **exception_reasons**: 异常原因记录

### 到期预警队列

- **正常 (绿色)**: 截止时间 > 24 小时
- **临期 (黄色)**: 剩余时间 ≤ 24 小时
- **逾期 (红色)**: 已超过截止时间

节点超时按责任人计算，逾期批量推进时逐条返回拦截结果。详情页保留补正动作、异常原因。

## 注意事项

1. **异常日志仅作为证据**，详情页的真实处理结果以提交为准
2. 队列刷新前自动校验原料订货材料、到货验收时限、库存回写证据
3. 所有操作均记录在 `processing_records` 表，可追溯完整操作历史
4. SQLite 数据库文件位于 `backend/data/orders.db`，如需重置可删除后重新执行 `npm run init-db` 和 `npm run seed`
