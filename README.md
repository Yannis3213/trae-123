# 养老护理院 - 月底集中处理照护记录系统

> 全流程闭环：**护登记员发起/补正 → 照护审核主管(护士长)办理 → 养老护理院复核负责人(院区主任)复核归档**

## 一、系统特性

| 模块 | 说明 |
| --- | --- |
| 角色 | 护登记员(REGISTRAR)、照护审核主管(AUDITOR/护士长)、养老护理院复核负责人(REVIEWER/院区主任) |
| 状态流转 | 待提交 → 待审核 → 待复核 → 审核通过 → 已同步 / 退回补正 |
| 业务区 | 照护记录登记（护登记员）、过程核验（护士长）、复核归档（院区主任） |
| 联动变化 | 药品发放新增、生命体征补正、异常上报复核结果 → 自动反映到照护记录状态/详情/版本号 |
| 后端校验 | 角色、当前处理人、状态、版本（乐观锁）、必填证据五重拦截；越权/重复提交/状态冲突/旧版本/缺证据一律拒绝 |
| 批量处理 | 批量提交、批量审核通过/退回、批量复核归档；结果逐条列出成功/失败原因 |
| 预警队列 | 正常（>2天）、临期（≤2天）、逾期 三色分明；逾期可手动推进提醒 |
| 持久化 | SQLite：照护记录、附件、处理记录、审计备注、异常原因、补正历史 |

## 二、端口约定（**请勿修改**）

| 服务 | 端口 | 配置位置 |
| --- | --- | --- |
| 前端 (Lit + Vite) | **3105** | `frontend/vite.config.js`  `server.port` |

> ⚠ 启动命令、请求地址、监听端口、CORS 白名单均共用上述端口，**无其他写死端口**。

## 三、启动方式

### 1. 后端 (Python 3.10+)

```bash
cd backend
python -m venv .venv && source .venv/bin/activate    # macOS / Linux
pip install -r requirements.txt
python main.py
# 启动后监听：http://0.0.0.0:8105
```

后端启动后可直接访问：
- 接口文档 (Swagger UI)：http://localhost:8105/docs

### 2. 前端 (Node.js 18+)

```bash
cd frontend
npm install
npm run dev
# 启动后访问：http://localhost:3105
```

## 四、SQLite 数据库初始化

**无需额外初始化命令**。

后端第一次启动时会自动：
1. 在 `backend/care_records.db` 创建 SQLite 文件；
2. 自动建表（users / care_records / attachments / process_records / audit_notes）；
3. 自动执行 `backend/seed.py` 种子脚本，写入 4 个演示账号 + 10 条演示照护记录 + 完整过程数据。

如需重置数据库：

```bash
# 停止后端后执行
rm backend/care_records.db
# 再次启动后端即自动重建
```

## 七、关键角色规则说明（后端强校验）

### 复核按钮（院区主任点击时）
1. ✅ **确认护士长是否已处理**：`auditor_id` 为空则拒绝 → `"护士长尚未完成审核处理"`
2. ✅ **院区主任是否具备复核权限**：非 REVIEWER 角色一律 403
3. ✅ **是否缺证据**：`missing_evidence` 非空则拒绝 → 记录继续留在待处理列表
4. ✅ **版本是否最新**：前端携带版本号与 DB 不一致 → 409 冲突
5. ✅ **状态是否允许**：仅 `PENDING_REVIEW` / `AUDITED_PASSED` 可复核
6. ✅ **逾期标记**：复核通过后仍保留 `overdue` 标记，审计备注自动留痕

### 审核按钮（护士长点击时）
1. 状态必须 `PENDING_AUDIT`
2. 版本号一致
3. 退回/通过会写入 `audit_remark` + 生成处理记录 + 缺失证据清单留痕
4. 通过后状态变更为 `PENDING_REVIEW`，版本号 +1

### 提交按钮（护登记员点击时）
1. 必须是本人创建 (`submitter_id == current_user.id`)
2. 状态必须为 `PENDING_SUBMIT` 或 `RETURNED`
3. 版本号必须匹配
4. 缺少证据会写入 `missing_evidence`，但不阻止提交（由护士长审核时判断）

## 八、刷新一致性

刷新页面后列表、详情、统计、操作记录、状态保持一致，因为：
- 所有状态变更均先写 DB，再返回结果；
- 版本号 (乐观锁) 防止覆盖更新；
- 逾期 (`overdue`) 字段在每次查询时按 `due_date` 重算并写回；
- 所有动作 (CREATE/SUBMIT/CORRECT/AUDIT/REVIEW…) 均追加写入 `process_records` 表，前端可还原完整历史。
