# 仓储配送中心 - 月底集中处理入库单系统

## 系统概述

面向仓储配送中心月底集中处理入库单的审批系统，包含 **库管员 → 仓储主管 → 运营经理** 三段审批流程，支持状态流转、证据留痕、异常补正、批量处理、到期预警和审计轨迹。

---

## 端口约定

| 服务 | 端口 | 地址 |
|------|------|------|
| 前端 (React + Rsbuild) | 3109 | http://localhost:3109 |
| 后端 (Rust + Rocket) | 8109 | http://localhost:8109 |

> **所有配置均使用上述端口**：前端 API 请求地址、后端 Rocket 监听端口、CORS 白名单已统一。

---

## 技术栈

- **前端**：React 18 + TypeScript + Rsbuild + Ant Design 5 + React Router 6 + Axios
- **后端**：Rust 1.75+ + Rocket 0.5 + rusqlite (SQLite) + serde + chrono
- **数据库**：SQLite（文件：`backend/data/warehouse.db`，首次启动自动创建并 seed）

---

## 启动步骤

### 1. 启动后端（端口 8109）

```bash
cd backend
cargo run
```

- 首次启动会自动：
  - 在 `backend/data/warehouse.db` 创建 SQLite 数据库
  - 执行 schema 初始化
  - 写入 seed 演示数据（用户、入库单、处理记录、审计备注、异常原因）

### 2. 启动前端（端口 3109）

```bash
cd frontend
npm install
npm run dev
```

打开浏览器访问 http://localhost:3109

## 三段状态流转

```
库管员提交
    ↓
【待确认】 → 仓储主管审核
                  ├─ 确认通过 → 【待确认】 → 运营经理最终确认 → 【已复查】（流程结束）
                  └─ 退回补正 → 【异常】   → 库管员补正后重新提交
```

| 状态标识 | 显示名称 | 含义 |
|----------|----------|------|
| `pending_confirmation` | 待确认 | 等待当前角色处理 |
| `exception` | 异常 | 被退回，等待补正 |
| `rechecked` | 已复查 | 运营经理最终确认，流程结束 |

## 三类业务模块

每张入库单包含三个可独立补正的业务模块，从详情页即可直接补正缺项：

| 模块 | 说明 | 证据字段 |
|------|------|----------|
| 📅 **入库预约** | 供应商到货预约信息 | `appointment_evidence` |
| 🔍 **质检上架** | 第三方质检报告与上架记录 | `inspection_evidence` |
| 📝 **入库单登记** | 仓库系统入库登记单据 | `registration_evidence` |

- 库管员提交前，三个模块必须全部完成（含证据）
- 被退回的模块可从详情页单独「补正」，无需整体重填
- 状态变更会带动模块完成标记、证据内容和异常原因联动更新

## 到期预警

运营经理视角可看到按预警分组的视图：

| 分组 | 判定条件 | 颜色 |
|------|----------|------|
| 🟢 **正常** | 距离截止 > 6 小时 | 绿色 |
| 🟠 **临期** | 距离截止 ≤ 6 小时 | 橙色 |
| 🔴 **逾期** | 当前时间 > 截止时间 | 红色 |

- 列表筛选支持按预警状态过滤
- 每张单据详情页顶部会显示预警提醒条
- **逾期的单据仅允许「补正」或「退回」操作**，后端会拒绝其他状态推进请求

## 后端权限与校验（接口层拦截）

调用 `/api/orders/process` 或 `/api/orders/batch` 时，后端按以下顺序逐一校验，任一不通过即返回错误：

| 校验项 | 说明 | 错误信息示例 |
|--------|------|-------------|
| **版本号** | 请求的 `version` 必须等于数据库最新版本 | `版本冲突，请刷新后重试` |
| **角色权限** | 当前登录用户角色必须等于单据 `current_handler_role` | `当前处理角色应为 仓储主管，您是 库管员，无权操作` |
| **处理人** | 如单据指定 `current_handler_id`，必须与登录用户一致 | `当前处理人应为...，您无权操作此单据` |
| **处理意见** | `opinion` 非空必填 | `处理意见必填` |
| **审计备注** | 运营经理操作时 `audit_note` 必填 | `运营经理必须填写审计备注` |
| **截止时间** | 已逾期单据仅允许补正/退回 | `该单据已逾期，仅允许补正或退回操作` |
| **角色动作匹配** | 各角色仅支持白名单内操作 | `库管员不支持操作: 最终确认` |
| **模块完整性** | 库管员提交前三个模块必须齐全 | `提交前需完成: 入库预约、质检上架` |
| **异常退回必填** | 退回必须填写 `exception_reason` 与 `exception_module` | `退回必须填写异常原因` |
| **状态冲突** | 非待确认状态不允许"确认通过/最终确认" | `仅待确认状态可确认通过` |

## 核心 API

所有接口需 `Authorization: Bearer <token>` 请求头。

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/login` | 登录，返回 token 与用户信息 |
| GET | `/api/orders?status=&urgency=` | 入库单列表（按当前角色过滤，运营经理可见全部） |
| GET | `/api/orders/<id>` | 入库单详情（含附件、处理记录、审计备注、异常原因、可用动作） |
| POST | `/api/orders/process` | 单张单据办理（状态变更） |
| POST | `/api/orders/batch` | 批量办理（逐条返回成功/失败与原因） |
| POST | `/api/orders/<id>/upload` | 上传附件登记 |

### 直接打接口测试示例

```bash
# 1. 登录获取 token
curl -X POST http://localhost:8109/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"keeper1","password":"123456"}'

# 2. 列入库单
curl http://localhost:8109/api/orders \
  -H "Authorization: Bearer u1:warehouse_keeper"

# 3. 越权测试（库管员尝试替主管操作主管的单据 o1，应返回 403/权限错误）
curl -X POST http://localhost:8109/api/orders/process \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer u1:warehouse_keeper" \
  -d '{"order_id":"o1","version":1,"action":"确认通过","opinion":"越权测试"}'
```

## 数据一致性

所有数据均写入 SQLite：

| 表 | 说明 |
|----|------|
| `users` | 用户与角色 |
| `inbound_orders` | 入库单主表（含状态、版本、三个模块证据、截止时间、上一处理意见/附件/审计） |
| `attachments` | 附件 |
| `processing_records` | 处理记录（审计轨迹，状态/角色/动作/意见/时间） |
| `audit_notes` | 审计备注 |
| `exception_reasons` | 异常原因与所属模块 |

刷新页面后，列表、详情、统计、操作记录均从数据库重新读取，结果一致。

## 批量处理

- 列表页勾选多条单据后点击「批量处理」
- 选择统一的操作类型（提交/补正/确认通过/最终确认/退回补正）
- 填写统一的处理意见、审计备注（运营经理）、异常原因（退回时）
- 提交后跳转「批量处理结果」页，**逐条显示成功/失败与具体原因**
- 后端逐条独立校验，失败不影响其他单据
