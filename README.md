# 银行网点 · 月底集中处理开户申请系统

基于三级审批责任链的开户申请月底集中处理系统。客户经理、运营主管、支行行长各司其职，各看各队列，状态全程可追溯。

## 技术栈

- **前端**: Next.js 14 (App Router) + TypeScript
- **后端**: Python Starlette (异步 ASGI)
- **数据库**: SQLite (本地文件)
- **认证**: JWT + bcrypt

## 端口约定

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 | 3002 | Next.js 开发服务器 |
| 后端 | 8002 | Starlette API 服务 |

> ⚠️ 前端请求地址、后端监听端口、CORS 白名单、启动命令均使用以上端口，请勿修改为其他端口。

## 快速开始

### 1. 安装依赖

```bash
# 后端依赖
cd backend
pip install -r requirements.txt

# 前端依赖
cd ../frontend
npm install
```

### 2. 初始化数据库

```bash
cd backend
python seed.py
```

执行后会在 `backend/data/bank.db` 创建 SQLite 数据库并插入演示数据。

### 3. 启动后端服务

```bash
cd backend
uvicorn main:app --host 0.0.0.0 --port 8002 --reload
```

后端 API 地址: http://localhost:8002

### 4. 启动前端服务

```bash
cd frontend
npm run dev
```

前端访问地址: http://localhost:3002

## 业务流程

### 三级审批责任链

```
客户经理提交 → 运营主管审核 → 支行行长复核 → 签收完成
     ↓              ↓              ↓
  补正重提 ←--- 退回补正 ←---- 退回补正
  (异常回传状态)
```

### 状态定义

| 状态 | 说明 |
|------|------|
| 待签收 | 等待对应角色签收处理（贯穿全流程） |
| 异常回传 | 退回客户经理补正材料 |
| 签收完成 | 支行行长复核通过，流程结束 |

### 各角色权限

#### 客户经理
- 查看自己名下的所有开户申请
- 处理异常回传的申请，补正后重新提交
- 查看处理记录、异常原因、审计轨迹

#### 运营主管
- 查看运营主管阶段的所有待处理申请
- 签收申请（声明由自己处理）
- 审核通过（流转到支行行长阶段）
- 退回补正（退回客户经理，记录异常原因）
- 批量处理
- 添加审计备注

#### 支行行长
- 查看支行行长阶段的所有待处理申请
- 签收复核
- 复核通过（流程终结，状态变为签收完成）
- 退回补正（退回客户经理）
- 批量处理
- 添加审计备注
- 查看已完成的申请

## 核心功能

### 列表与筛选
- 按角色展示对应队列
- 按状态、到期状态筛选
- 按客户名、申请编号、电话搜索
- 各角色只看自己该看的

### 详情办理
- 基本信息、账户信息、状态版本
- 处理记录时间线（审计轨迹）
- 异常原因列表（未解决高亮）
- 审计备注
- 附件材料
- 根据角色和状态动态显示可执行操作

### 批量处理
- 多选申请批量操作
- 逐条独立校验（角色、状态、版本、处理人）
- 每条返回成功/失败及具体原因
- 失败不影响其他条目的成功
- 支持批量签收、批量审核通过、批量退回补正

### 到期预警
- 三队列独立展示：正常 / 临期（3天内） / 逾期
- 显示剩余/逾期天数
- 显示节点超时责任人
- 逾期批量推进逐条拦截

### 版本与状态冲突
- 每次操作版本号 +1
- 提交时校验版本，旧版本提交直接拦截
- 状态冲突时保留原值，提示需要谁补正
- 越权操作、重复提交均被拦截

### 审计与责任追溯
- 所有操作均记录处理人、角色、时间、前后状态、前后版本
- 审核操作记录证据清单
- 退回操作记录异常原因与补正要求
- 审计备注独立记录，仅运营主管及以上可添加

## 数据库表结构

### account_applications (开户申请表)
- 申请编号、客户信息、账户类型、金额
- 状态、当前处理人、当前阶段角色、版本号
- 客户经理、支行、到期日期
- 创建时间、更新时间

### attachments (附件表)
- 关联申请、文件名、类型、上传人

### processing_records (处理记录表)
- 操作类型、前后状态、前后版本
- 操作人、操作角色、备注
- 证据要求、证据提供
- 操作时间

### exception_reasons (异常原因表)
- 异常类型、描述
- 上报人、是否已解决、解决人

### audit_notes (审计备注表)
- 备注内容、备注人、备注角色

### users (用户表)
- 用户名、密码哈希、真实姓名、角色、支行

## API 接口

### 认证
- `POST /api/auth/login` - 登录
- `GET /api/auth/me` - 获取当前用户

### 开户申请
- `GET /api/applications` - 列表（按角色自动过滤队列）
- `GET /api/applications/stats` - 统计数据
- `GET /api/applications/:id` - 详情（含记录、异常、备注、附件）
- `POST /api/applications/:id/process` - 办理操作
- `POST /api/applications/batch` - 批量处理
- `POST /api/applications/:id/audit-note` - 添加审计备注

### 列表查询参数
- `status`: 状态筛选
- `due_status`: 到期状态筛选
- `queue`: 队列筛选 (my/returned/completed)
- `keyword`: 关键词搜索

## 常见问题

### Q: 如何重置数据库？
A: 删除 `backend/data/bank.db` 文件，重新运行 `python seed.py` 即可。

### Q: 如何测试状态冲突？
A: 打开两个标签页登录同一角色，同时进入同一申请详情。在第一个标签页执行操作后，第二个标签页再操作会提示版本冲突。

### Q: 逾期的责任人怎么认定？
A: 根据当前处理人和当前阶段认定：
- 未签收：待分配（对应角色负责人）
- 已签收：签收人
- 异常回传：客户经理

### Q: 批量处理中部分失败怎么办？
A: 成功的会正常更新状态和版本，失败的会返回具体原因。可根据失败原因修正后单独处理。
